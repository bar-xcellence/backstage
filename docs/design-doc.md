## Problem Statement

Murdo MacLeod, owner of Bar Excellence, creates event briefs manually in Word documents. Each brief takes 30-60 minutes to type from scratch. He then emails them to Rory at Liquor Collective (LC), their bartending partner. This manual process has caused real-world errors, including a missed NEC event where bartender kit details were forgotten, requiring an emergency drive back to site. Backstage replaces this with structured data entry and one-button "Send to LC" dispatch.

## Demand Evidence

- One committed customer: Murdo MacLeod, who will use this for every event
- Real cost of current process: 30-60 minutes per brief, multiplied by every event
- Real error with real consequences: the NEC incident (missed equipment, emergency drive)
- Events are coming up — every week without Backstage is another manual brief
- Revenue model: consulting/CTO relationship, not SaaS. Backstage is part of the value Rob delivers to Bar Excellence.

## Status Quo

Murdo's current workflow:
1. Opens a blank Word document
2. Types every detail from scratch — venue, times, cocktail specs, equipment, install instructions
3. Manually calculates stock quantities
4. Emails the Word doc to Rory at LC
5. Hopes nothing was forgotten

Cost: 30-60 min per brief. No templates, no reuse, no safety net. The NEC incident proves the failure mode is real.

## Target User and Narrowest Wedge

**Primary user:** Murdo MacLeod, owner of Bar Excellence. Manages LC partnership events. Success = flawless events, happy clients. Failure = missed details, embarrassing errors in front of clients.

**Secondary user:** Rory at LC Group. Non-technical. Receives event briefs. Needs clear, complete information to prepare staff and stock.

**Narrowest wedge:** The "Send to LC" button. Event CRUD with cocktail selection, stock calculator, and one-button brief generation (email + PDF). This is the feature that saves Murdo 30-60 minutes per event and eliminates the class of errors that caused the NEC incident.

## Premises

1. **Auth needs full brand control.** WorkOS AuthKit uses hosted login pages. For a charcoal/gold/Cormorant Garamond branded login with 3 users, a self-hosted auth solution (Auth.js v5 with Resend magic links) gives full pixel control at zero cost.

2. **Phase the 13-table schema.** The core loop (events, cocktails, ingredients, garnishes, event_cocktails) ships first. Equipment templates, standard notes, checklists layer in during later phases.

3. **"Send to LC" is the killer feature.** Not the dashboard, not the kanban. The thing that saves 30-60 minutes per event and eliminates errors.

4. **Light theme only at launch.** Ship one theme (cream-based, matching mockups). Dark mode is a Phase 3+ enhancement.

5. **Reserve Noir design system is production-ready.** The 0px radius, charcoal/cream/gold, Cormorant Garamond + Raleway, ghost borders, tonal layering — implement as documented.

### Auth Architecture: Branded Login with Auth.js v5

**Why Auth.js v5 over WorkOS AuthKit:**
- Full pixel control over login page (Reserve Noir branded: charcoal background, gold accents, Cormorant Garamond heading, Raleway inputs)
- Magic link via Resend (already in stack for "Send to LC")
- Zero cost for 3 users (no per-MAU pricing concerns ever)
- Custom JWT claims for role-based access (owner, super_admin, partner)
- Session management via secure HTTP-only cookies

**Three-layer security model (preserved from original PRD):**
1. Middleware: redirects unauthenticated users
2. Server Actions: verify session + check role
3. NeonDB RLS: PostgreSQL policies per role

**RLS implementation with Neon serverless driver:**
The `@neondatabase/serverless` HTTP driver executes each query as an implicit transaction. Use `neon.transaction()` to wrap SET LOCAL + query in the same transaction block:

```typescript
import { db } from '@/db';
import { sql } from 'drizzle-orm';

async function withRLS<T>(role: string, userId: string, queryFn: (tx: typeof db) => Promise<T>): Promise<T> {
  return db.transaction(async (tx) => {
    await tx.execute(sql`SET LOCAL "app.user_role" = ${role}`);
    await tx.execute(sql`SET LOCAL "app.user_id" = ${userId}`);
    return queryFn(tx);  // pass tx so queries execute within the transaction
  });
}
```

**Column-level financial data exclusion:** PostgreSQL RLS operates on rows, not columns. Financial columns (invoice_amount, cost_amount, profit_amount, card_payment_*) are excluded from the partner role via a database VIEW:

```sql
CREATE VIEW events_partner WITH (security_invoker = true) AS
  SELECT id, event_name, show_name, event_date, venue_name,
         arrive_time, service_start, service_end, depart_time,
         guest_count, prepaid_serves, station_count, staff_count,
         status, notes_custom
  FROM events;
-- security_invoker = true (Postgres 15+) ensures RLS on the underlying
-- events table is evaluated using the querying role, not the view owner

GRANT SELECT ON events_partner TO partner;
-- RLS policy on the view filters to confirmed+ events
```

## Stock Calculator Specification

Algorithmic calculation, no AI required.

**Formula per event:**
```
total_serves = prepaid_serves + card_payment_serves (if applicable)

For each cocktail on the event menu:
  allocated_serves = event_cocktails.serves_allocated
                     OR (total_serves / number_of_cocktails)  -- even split if not overridden

  For each ingredient in that cocktail:
    total_ml = allocated_serves x ingredient.amount  (in ingredient.unit)
    with_buffer = total_ml x 1.15  (15% wastage buffer for spirits/mixers)
    purchase_units = ceil(with_buffer / standard_bottle_size)
    -- standard_bottle_size: application-level constants (not database-stored)
    -- 700ml for spirits (UK standard), 1000ml for juices/syrups, 500ml for purees
    -- Regional note: UK uses 700ml bottles; if US events arise (750ml), update constants

  For each garnish:
    total_garnishes = allocated_serves x garnish.quantity
    with_buffer = total_garnishes x 1.10  (10% garnish buffer)
```

**Aggregation:** Ingredients with the same name across multiple cocktails are summed before the purchase_units calculation. Example: if Cocktail A needs 30ml vodka/serve and Cocktail B needs 45ml vodka/serve, with 100 serves each = (3000 + 4500) x 1.15 = 8625ml = 13 bottles (700ml).

**Worked example:**
- Event: 200 guests, 4 cocktails, even split = 50 serves each
- Cocktail "Espresso Martini": vodka 50ml + coffee liqueur 25ml + espresso 30ml
- Vodka: 50 x 50 = 2500ml x 1.15 = 2875ml = 5 bottles (700ml)
- Coffee liqueur: 50 x 25 = 1250ml x 1.15 = 1438ml = 3 bottles (700ml)
- Espresso: 50 x 30 = 1500ml x 1.15 = 1725ml = 2 bottles (1000ml)

**Output:** Grouped by ingredient_category (spirits, juices, syrups, purees, garnishes), sorted alphabetically within each group. Shows: ingredient name, brand (if specified), total quantity needed, purchase units.

## Equipment Template Scaling Rules

Three template types, each scales per station (1 bartender = 1 station = 1 kit set):

| Scaling Rule | Meaning | Example |
|---|---|---|
| per_station | Multiply base_quantity by station_count | 1 shaker x 3 stations = 3 shakers |
| fixed | Same quantity regardless of stations | 1 first aid kit = 1 first aid kit |
| per_spirit | Multiply by number of distinct spirits on menu | 1 store n pour per spirit |
| per_ingredient | Multiply by number of distinct ingredients | 1 garnish box per garnish type |
